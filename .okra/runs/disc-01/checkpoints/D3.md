# D3 checkpoint — nats.ws browser consumer
- Browser/WS client connects `ws://localhost:8081`, jetstream() over WS works, ordered consumer streams events — proven twice: headless Node using the exact browser import path, and a real Chromium tab via agent-browser/CDP.
- Correct imports (Context7-confirmed against `/nats-io/nats.js`):
  - `wsconnect` → `@nats-io/nats-core` (NOT `@nats-io/transport-node` — that package's own docs snippet is misleading; the "Running in the Browser" pages explicitly say `wsconnect()` is exported from `@nats-io/nats-core` for browser/Deno/Node-22-native-WS use).
  - `connect` (TCP, Node) → `@nats-io/transport-node`.
  - `jetstream` (client + `js.consumers.get(stream)` for an ordered/ephemeral consumer, `js.publish(subject, payload, {msgID})`) → `@nats-io/jetstream`.
  - The task's given skeleton code already had these three right; no fix was needed.
- Decision: browser uses an EPHEMERAL ordered consumer (`js.consumers.get('VIDGEN_EVENTS')`, no name arg) — no server-side durable state per tab. Ordered consumers auto-recreate on gaps/inconsistency and don't need real acking (calling `m.ack()` is harmless but effectively a no-op under the hood).
- Evidence:
  - Headless WS consume (`ws-check.ts`, same `wsconnect` import as the browser): received all 3 freshly-published test events plus backlog (4/4, then re-run 3/3) over `ws://localhost:8081`. PASS.
  - Browser DOM check (`agent-browser` against `npx vite` on `http://localhost:5183`): opened the page, `<ul id="log">` rendered the full backlog on load; with the tab left open and idle, ran `publish-live.ts` and polled the DOM — item count went from 10 → 13 (`<li>` items with the fresh `nonce`) with **no page reload**, confirming genuine live append. Browser console clean (no errors) on both initial load and after a fresh reload replaying all 10 backlog events instantly.
- Latency observed:
  - Node (WS transport, same import path as browser): publish→callback receipt was **1–5ms** (`latency-check.ts`, unique msgIDs to dodge JetStream dedup).
  - Real-browser DOM: publish call (via `npx tsx`, ~850ms of which is Node/tsx cold-start) to DOM count reflecting the 3 new items was **≤260ms** after the publish script itself returned — well under the 1s target. NATS/WS delivery is not the bottleneck; script cold-start dominates.
- Open: WS auth for non-loopback deploy (defer — v1 loopback only, matches NATS setup used here); reconnect/backoff in the store (not exercised — `wsconnect`'s default reconnect behavior wasn't stress-tested); `ws-check.ts`/`latency-check.ts` call `process.exit()` instead of draining `c.consume()`'s returned promise cleanly — fine for a spike script, but the real store needs a proper unsubscribe/drain path on unmount.
- Confidence: HIGH — 3+ events consumed over WS in both headless Node and a real browser tab, live-appending into the DOM with sub-second latency and no console errors.
