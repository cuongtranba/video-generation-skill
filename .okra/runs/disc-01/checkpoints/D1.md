# D1 checkpoint — Agent SDK integration

- Agent SDK confirmed to wrap the local `claude` CLI under subscription (OAuth) auth: no `ANTHROPIC_API_KEY` was ever set in the shell env, and `~/.claude.json` shows `oauthAccount.billingType: "stripe_subscription"`, `organizationType: "claude_max"`, `organizationRateLimitTier: "default_claude_max_5x"`. `claude --help` confirms the *default* mode (what `query()` uses) reads OAuth/keychain; only `--bare` mode forces `ANTHROPIC_API_KEY`-only auth, which we did not use. So the auth-mechanism half of the "CRITICAL FACT" is verified correct.
- **`total_cost_usd` is NOT $0 — FLAG.** Observed real values: idea 1 = `$0.214275`, idea 2 = `$0.379934`, idea 3 = `$0.143099` (single-idea deep-dive rerun of idea 1 = `$0.334613`, showing run-to-run variance). These are material relative to the project's own `$0.15` per-video `COST_CAP_USD` anti-goal cap.
- Root cause (confirmed via full `result` message inspection, not guessed): `total_cost_usd` is computed from `modelUsage["claude-opus-4-8"].costUSD`, itself derived from raw token counts (`inputTokens: 16481`, `outputTokens: 2363`, `cacheReadInputTokens: 22286`, `cacheCreationInputTokens: 18199`) multiplied by published per-token API list pricing. This is a **notional/telemetry cost estimate**, not an actual charge — under a flat-rate Claude Max subscription there is no per-token billing, so the true marginal cost of these calls is $0 (subject to the subscription's rate-limit tier, not a dollar meter). The field exists for cost-observability parity with API-key users; it does not reflect what was actually billed here.
- Secondary finding: `query()` with no `model` option defaulted to **`claude-opus-4-8`** (1M context, 64K max output) — the priciest tier — which is why the notional cost is as high as it is (Opus-tier token rates × ~35–40K total tokens including cache writes). Pinning a cheaper model would lower the *notional* number further but doesn't change the *real* subscription cost, which stays $0 either way.
- Structured output works correctly: all 3/3 ideas produced valid `structured_output.scenes` arrays matching the JSON schema, 3 scenes each (in range). The confirmed working `query()` shape:
  ```typescript
  query({
    prompt: '...',
    options: { outputFormat: { type: 'json_schema', schema } },
  })
  ```
  Cost and structured output are both read off the terminal `message.type === 'result'` message: `message.total_cost_usd` (number) and `message.structured_output` (typed per your schema).
- Exact 3 probe readings (script-cost.ts, `npx tsx script-cost.ts`, no `ANTHROPIC_API_KEY` in env):
  ```
  {"idea":"3 lý do bạn nên uống nước ấm mỗi sáng","scriptUsd":0.214275,"scenes":3}
  {"idea":"5 mẹo tiết kiệm pin điện thoại","scriptUsd":0.37993350000000004,"scenes":3}
  {"idea":"Cách pha cà phê phin ngon tại nhà","scriptUsd":0.143099,"scenes":3}
  ```
- Total per-video *real* cost = $0 (script, subscription-billed) + ~$0.004 (FPT TTS) + $0 (render) ≈ $0.004. `COST_CAP_USD` default $0.15 gives ~37x headroom on real spend — **but only if the webapp's cost ledger is built to treat SDK `total_cost_usd`/`modelUsage.costUSD` as notional and exclude it from the real per-video dollar sum** for subscription-authenticated runs. If the ledger naively sums `total_cost_usd` as real spend, a single script-gen call (~$0.14–$0.38 observed) would itself blow past the $0.15 anti-goal cap on phantom cost — a real design risk to flag for the webapp rewrite's cost-wall component (`internal/cost` equivalent).
- Confidence: **FLAG** — not because the SDK/subscription mechanism is broken (it works, and structured JSON output is solid, 3/3), but because the task's assumption that `total_cost_usd` reads "$0 / very near 0" is empirically false, and the webapp rewrite's cost-tracking design must explicitly account for the notional-vs-real distinction rather than trusting the SDK's reported field at face value.
